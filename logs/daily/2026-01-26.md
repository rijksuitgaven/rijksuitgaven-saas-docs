### 9. Query Performance Optimization

**Problem:** Aggregation queries took 9-19 seconds (target: <500ms)

**Root Cause:** GROUP BY on 674K rows requires full table scan even with index

**Solution:** Created materialized views (pre-computed aggregations) for all 6 modules:

| View | Source Table | Primary Field |
|------|--------------|---------------|
| `instrumenten_aggregated` | instrumenten | ontvanger |
| `apparaat_aggregated` | apparaat | kostensoort |
| `inkoop_aggregated` | inkoop | leverancier |
| `provincie_aggregated` | provincie | ontvanger |
| `gemeente_aggregated` | gemeente | ontvanger |
| `publiek_aggregated` | publiek | ontvanger |

**Performance Results:**

| Module | Before | After | Improvement |

### 11. Year Filter Design Decision (ADR)

**Decision:** Option B - Filter on aggregated view (users always see trends)

When filtering by year (e.g., `jaar=2024`):
- Shows recipients who have data in that year
- Still returns ALL years in response (2016-2024)
- Users can see the full trend, filtered to recipients active in selected year

Implementation: `WHERE "2024" > 0` on aggregated view

### 14. Search Performance Optimization (pg_trgm + GIN indexes)

Added trigram indexes for faster ILIKE searches:

```sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;
CREATE INDEX ... USING gin (column gin_trgm_ops);
```

**Results (3+ character searches):**

| Module | Before | After | Improvement |
|--------|--------|-------|-------------|
| Integraal 'stichting' | 970ms | 226ms | 4.3x faster |
| Inkoop 'ministerie' | 760ms | 190ms | 4x faster |
| Instrumenten 'prorail' | ~700ms | 237ms | 3x faster |
| All modules | 600-1500ms | **175-600ms** | ✅ Under target |

**Note:** Single-character searches still slow (expected) - Typesense handles those.
